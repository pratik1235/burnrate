Create an integration test suite which basically does the end to end test. for a given input directory path the test suite should scan all statement files inside the directory and run the tests described in the below section.

input : directory path where the statement files are.

environment - this test suite would use a new environment i.e a new database and would not touch the application's main database for testing.

test case scenarios :


| No. | Title | Description | Acceptance Criteria |
| --- | --- | --- | --- |
| 1. | Test Statement Imports from watch folder | All statement files inside the watch folder should be recursively imported into the application. watch folder would be same as the input statement directory. | Match the total number of files with total number of statement records created in the database. in case of federal bank scapia statement files - there would be 2 statement rows created per statement file. |
| 2. | Test Statement uploads by selecting folder from drop zone | All statement files inside the input statement directory should be recursively imported into the application. | Match the total number of files with total number of statement records created in the database. in case of federal bank scapia statement files - there would be 2 statement rows created per statement file. |
| 3. | Test single statement upload from Statements page | Upload one valid statement file through the UI drop zone or file picker. | API returns success, exactly one statement row is created unless the file is a multi-statement format, all expected transaction rows are created, and the Statements page shows the imported statement without a page refresh. |
| 4. | Test recursive import with nested folders | Use an input directory containing statement files at multiple folder depths along with empty folders. | Every valid statement file in every nested folder is processed exactly once, empty folders are ignored, and statement `file_path` or `original_upload_path` preserves enough path information to identify the source file. |
| 5. | Test unsupported files are skipped | Include non-statement files such as txt, png, json, temporary files, and extensionless files in the input directory. | Unsupported files do not create statement, transaction, or processing log success rows; the bulk upload response records them as rejected or skipped with an invalid type reason. |
| 6. | Test mixed valid and invalid bulk upload | Upload a folder containing valid PDFs, valid CSVs, invalid files, corrupted PDFs, and oversized files. | Valid files are imported, invalid files are reported independently, the whole bulk job does not fail because one file failed, and response counts match success, failed, rejected, duplicate, parse_error, password_needed, and card_not_found totals. |
| 7. | Test file size limit | Upload or watch-import a statement file larger than the configured maximum upload size. | The request fails or file is rejected with a file too large error, no statement or transaction rows are created, and the UI shows a clear failure state. |
| 8. | Test duplicate statement upload | Import the same statement file twice through the same flow. | The second import returns duplicate, does not create additional statement or transaction rows, and the existing statement data remains unchanged. |
| 9. | Test duplicate across upload and watch folder | Import a file through the watch folder and then upload the same file manually, and also test the reverse order. | Duplicate detection uses file hash and card identity consistently across both flows, and only the first successful import creates database records. |
| 10. | Test concurrent duplicate import race | Trigger two imports of the same file at the same time, for example by uploading while the watch folder also detects it. | At most one successful statement record is committed, duplicate or integrity handling is graceful, no orphan transactions are created, and the UI does not show contradictory success messages. |
| 11. | Test Federal Bank Scapia split statement import | Import representative Federal Bank Scapia statement files. | Each Scapia source file creates exactly two statement rows when expected, both rows share the correct file hash behavior, and transactions are split into the correct statement/card contexts. |
| 12. | Test credit card PDF parser coverage by bank | Import known-good fixture statements for each supported credit card parser such as HDFC, ICICI, Axis, Federal, Indian Bank, and any additional supported banks present in fixtures. | For each fixture, statement bank, card_last4, period_start, period_end, transaction_count, total_spend, total_amount_due, credit_limit, currency, and payment_due_date match expected fixture values. |
| 13. | Test bank account CSV parser coverage | Import supported bank account CSV files and generic auto-detected CSV fixtures. | Statement rows have `source = BANK`, bank account last4 is populated where available, transaction rows have `source = BANK`, and bank account transactions are visible through BANK filters. |
| 14. | Test mixed credit card and bank account imports | Import CC PDF statements and BANK CSV statements in the same bulk job. | The application stores and displays both sources correctly, CC statements do not appear as bank accounts, BANK statements do not require credit card records, and source filters separate or combine them correctly. |
| 15. | Test automatic bank detection | Import files without passing a bank override and with names/content that should identify the issuing bank. | The detected statement bank matches the expected fixture bank, parser-specific fields are extracted, and detection failures are reported as parse errors rather than incorrect successful imports. |
| 16. | Test manual bank override | Import a statement while selecting or passing an explicit bank override. | The selected bank parser is used, the stored statement bank matches the override when parsing succeeds, and a wrong override produces a controlled parse_error or card_not_found result. |
| 17. | Test card resolution for registered cards | Configure cards matching the fixture statement card_last4 values before import. | Imported CC statements link transactions to the correct card_id, transaction bank and card_last4 match the statement, and multi-card imports do not cross-link transactions to the wrong card. |
| 18. | Test card not found flow | Import a CC statement whose card_last4 is not configured. | The import returns card_not_found or the expected failure status, no successful statement with transactions is created for that card, and the UI guides the user to add the missing card. |
| 19. | Test encrypted statement auto-unlock | Import encrypted statements whose password can be generated from configured user name, DOB, or card details. | The file is unlocked without manual password entry, statement and transaction records are created, and no password_needed status remains for that file. |
| 20. | Test encrypted statement manual password retry | Import an encrypted statement that cannot be auto-unlocked, then retry with the correct password from the UI. | Initial import creates a password_needed state, retry succeeds, the original failed row is replaced or updated according to app behavior, and final statement/transaction data matches expected fixture values. |
| 21. | Test wrong password retry | Retry a password_needed statement with an incorrect password. | The statement remains password_needed or returns a clear failure, no partial transactions are committed, and retry with the correct password still succeeds afterwards. |
| 22. | Test malformed or truncated statement file | Import corrupted PDFs, malformed CSVs, blank PDFs, and files with no transaction table. | Each invalid file creates a parse_error or controlled failure with status_message, no uncaught backend exception reaches the test runner, and the Statements page can filter and display failed rows. |
| 23. | Test statement metadata correctness | Validate every imported statement row against fixture expectations. | `statements.bank`, `card_last4`, `period_start`, `period_end`, `file_hash`, `file_path`, `original_upload_path`, `transaction_count`, `total_spend`, `total_amount_due`, `credit_limit`, `source`, `status`, `parse_failed`, `currency`, and `payment_due_date` are exactly correct or within documented rounding tolerance. |
| 24. | Test transaction data correctness | Validate every transaction row created from each fixture. | `transactions.statement_id`, `date`, `merchant`, `amount`, `type`, `category`, `description`, `bank`, `card_last4`, `card_id`, `source`, and `currency` match expected fixture data, and every transaction belongs to exactly one statement. |
| 25. | Test statement totals reconcile with transactions | Compare parsed statement totals against transaction rows for each imported file. | Statement `transaction_count` equals the number of child transactions, `total_spend` matches the expected sum of debit transactions according to app rules, credits are handled correctly, and totals are rounded consistently to two decimals. |
| 26. | Test credit, refund, fee, tax, and payment transaction classification | Import fixtures containing purchases, refunds, card payments, finance charges, taxes, annual fees, reversals, and cashbacks. | Transaction `type`, amount sign, category, and aggregate contribution match expected behavior, including excluding card payment transactions from spend metrics where applicable. |
| 27. | Test Indian Bank repayment handling | Import Indian Bank fixtures containing repayment or credit card bill payment rows. | Repayment rows are categorized as cc_payment or otherwise excluded from spend totals as expected, while regular spend rows remain included. |
| 28. | Test multi-currency imported data | Import statements containing non-INR currency values or configured display currency cases. | Statement and transaction `currency` values are stored as uppercase ISO codes, analytics groups or suppresses single-currency totals correctly, and no currency is silently mixed into INR totals. |
| 29. | Test file hash and path persistence | Import fixtures with similar filenames, duplicate names in different folders, and special characters in paths. | File hashes match the SHA-256 of the stored content, duplicate names do not collide, stored filenames are sanitized, and UI display paths show original client paths for manual uploads and watch paths or basenames for watcher imports. |
| 30. | Test path traversal safety | Attempt uploads with filenames containing path traversal, absolute paths, Windows separators, and unusual unicode or whitespace. | Files are persisted only under the allowed upload directory, no outside file is created or overwritten, and display paths do not expose unsafe server paths for manual uploads. |
| 31. | Test database isolation for e2e suite | Run the full test suite using a dedicated test database and temporary uploads/watch directories. | No records are written to the application's main database, the suite can be rerun from a clean state, and teardown removes or isolates all test-created files. |
| 32. | Test import rollback on failure | Force parser failure after a statement row starts processing, for example with a fixture that fails mid-parse. | No partial successful statement or orphan transaction rows are left behind unless the expected behavior is a parse_error statement row with zero transactions and a status message. |
| 33. | Test delete statement cascade | Delete an imported statement through the API and through the UI. | The statement row is removed, child transactions and transaction_tags are removed, unrelated statements remain intact, and aggregate dashboard/analytics values update after deletion. |
| 34. | Test delete shared duplicate resources | Delete one statement when another statement shares the same file hash, such as split Scapia statements. | Shared file/password metadata is kept while another statement still uses it, and removed only when the final related statement is deleted. |
| 35. | Test single statement reparse | Reparse one imported statement from its stored file path. | Existing rows for that statement are replaced with freshly parsed data, no duplicate child transactions remain, statement metadata stays correct, and UI shows the updated counts and status. |
| 36. | Test reparse all statements | Reparse all imported statements after a bulk import. | The endpoint processes valid stored file paths, skips missing files, limits concurrency as expected, returns accurate success/failed/skipped counts, and final database counts match the original fixture expectations. |
| 37. | Test reparse while import is running | Start a bulk import and trigger reparse all or single reparse before import completion. | The application avoids deadlocks and duplicate rows, returns clear statuses, and final data is internally consistent after all background work completes. |
| 38. | Test high-volume bulk import performance | Import a large realistic directory containing many months, banks, cards, and sources. | Processing completes within the agreed performance threshold, concurrency does not exceed the configured worker limit, database writes remain successful under WAL/concurrent reads, and all counts reconcile. |
| 39. | Test processing logs for watch and bulk imports | Import successful, duplicate, password_needed, parse_error, card_not_found, and failed files. | ProcessingLog rows contain correct file_name, status, message, bank, transaction_count, acknowledged, and created_at values; the API returns unread logs in newest-first order and limits results appropriately. |
| 40. | Test processing log acknowledgement | Acknowledge one or more processing logs from the UI or API. | Acknowledged logs no longer appear when unread_only is true, still appear when unread_only is false if supported, and acknowledging a missing log is harmless or returns the documented result. |
| 41. | Test processing toast and polling UI with Playwright | Use Playwright to import files through the watcher or upload flow while the app is open. | Toasts or processing notifications appear for new logs, duplicate toasts are not shown repeatedly after acknowledgement, and the UI updates without a manual page reload. |
| 42. | Test Statements page source filters with Playwright | After mixed CC and BANK imports, filter the Statements page by source. | CC-only, BANK-only, and all-source views show the correct rows and counts, and filter state survives normal navigation where intended. |
| 43. | Test Statements page bank and date filters with Playwright | Use bank, from date, and to date filters on the Statements page. | Rows match period-overlap filtering rules, multi-bank filters work, no matching rows show an empty state, and clearing filters restores all rows. |
| 44. | Test Statements page search with Playwright | Search by bank, card_last4, source, filename/path, status, and status message. | Search results match the backend query behavior, search is case-insensitive where expected, and parse_error/password_needed rows are searchable. |
| 45. | Test Statements page sort and pagination with Playwright | Sort statements by default import order, amount due, due date, and transaction count with ascending and descending orders. | Rows are ordered correctly with password_needed and parse_error priority behavior preserved, pagination does not duplicate or skip rows, and total counts remain accurate. |
| 46. | Test Statements page password-needed UI with Playwright | Import an encrypted statement that requires manual password and complete the inline retry flow. | The row shows a lock/password-needed state, entering the correct password reparses successfully, the row changes to success, and transactions become available. |
